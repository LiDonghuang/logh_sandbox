# v5.0-alpha1 — Contact Hysteresis (CH) Execution Spec (Engineering-only)

**Authority:** Epoch II Governance  
**Scope:** Phase V minimal primitive — *Contact Hysteresis*  
**Implementation status:** Authorized for **alpha experiment** (constrained)  
**Non-goals:** No movement-objective change, no decision logic, no new personality dimensions, no mapping edits.

---

## 0) Glossary

- `attack_range` (**existing**): physical maximum weapon reach (must not be exceeded).
- `r_enter`: stricter threshold to *enter* contact (must be ≤ `attack_range`).
- `r_exit`: threshold to *exit* contact (must be == `attack_range` in alpha).
- `h`: hysteresis band ratio, **bounded**.

---

## 1) Exact Definition (pseudocode)

### 1.1 Constants / Parameters (alpha)

- `h ∈ [0.0, 0.2]` (hard cap)
- `r_exit = attack_range`
- `r_enter = attack_range * (1 - h)`

> This guarantees: **CH never expands** effective firing range beyond physical `attack_range`.

### 1.2 Per-unit state (allowed)

Introduce **only** the following internal runtime states (not personality parameters):

- `engaged: bool` (default `False`)
- `engaged_target_id: int | None` (default `None`)

No group-level memory. No fleet-level memory. No global history buffers.

### 1.3 Contact evaluation (core)

Assume each unit already has a selected `target_id` from existing deterministic targeting.

Let `d = distance(unit, target)`.

```
if unit.engaged is True and unit.engaged_target_id == target_id:
    # maintain contact until beyond r_exit
    in_contact = (d <= r_exit)
else:
    # entering contact requires stricter r_enter
    in_contact = (d <= r_enter)

# update state AFTER computing in_contact
if in_contact:
    unit.engaged = True
    unit.engaged_target_id = target_id
else:
    unit.engaged = False
    unit.engaged_target_id = None
```

### 1.4 Interaction with targeting (strict)

- Target selection logic **must not** read `engaged` or `engaged_target_id`.
- If your engine allows target switching each tick, CH state resets naturally when target differs (as above).
- Tie-breakers remain **exactly** as baseline (index-based, mirror-safe).

### 1.5 Damage application (unchanged)

Damage is applied **iff** `in_contact` is True.  
All other parts of the damage model remain unchanged.

## 4) Tick Insertion Point

CH must be inserted **only** at the point where the engine currently does:

1) snapshot state read  
2) deterministic targeting (unchanged)  
3) **range / contact evaluation**  ← **insert CH here**  
4) damage resolution (unchanged)  
5) removal / bookkeeping (unchanged)  
6) movement update (unchanged)

No other insertion points permitted.

## 5) Required Verification Suite (minimum)

### 5.1 Determinism

- Run the *same* scenario twice with identical initial conditions.
- Assert full trajectory equivalence (or strict per-tick aggregate equality):
  - remaining units per side per tick
  - total HP per side per tick
  - battle end tick

### 5.2 Mirror symmetry (Level 1)

- Run mirrored initial conditions.
- Track antisymmetry metric (your existing one) and confirm it does **not worsen** vs baseline.

### 5.3 Backward-compatibility regression

- Set `h = 0.0` and confirm results match Phase IV baseline (within numeric tolerance you already use).

### 5.4 Extreme parameter stress

At minimum, run these grids (no randomness, fixed seeds not needed):

- ODW ∈ {0.0, 1.0}
- FR ∈ {0.0, 1.0}
- Mobility extreme scenario (whatever your harness uses to represent max movement)
- h ∈ {0.0, 0.10, 0.20}

### 5.5 Oscillation / boundary jitter check

- Detect rapid flip-flopping of `in_contact` for the same unit-target across consecutive ticks.
- If present, report counts and the distance histogram near thresholds.

### 5.6 Duration distribution

- Report battle duration (tick count) across the scenario set:
  - baseline vs h=0.10 vs h=0.20
