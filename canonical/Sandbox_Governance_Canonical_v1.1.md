# Sandbox Governance Canonical

## Version 1.1

Status: Active
Authority: Governance Core (App Layer)

---

# I. Governance Scope

This document defines:

* Phase control authority
* Structural invariants
* Evolution boundaries
* Engineering guardrails
* Cross-layer discipline

Governance owns:

* Archetype Anchor interpretation
* Mapping integrity
* Phase escalation decisions
* Stability guardrails
* Canonical documentation authority

Engineering owns:

* Implementation
* Numerical model execution
* Debugging
* Runtime optimization

Governance overrides Engineering in structural disputes.

---

# II. Structural Invariants (Non-Negotiable)

The following must remain true across all Phases:

1. Determinism (no randomness unless explicitly authorized)
2. Mirror Symmetry Level 1 preserved
3. Backward compatibility where specified
4. No silent parameter drift
5. No implicit cross-layer coupling
6. No architecture refactor without Gate approval

Violation triggers immediate freeze review.

---

# III. Phase Gate Discipline

Before Phase escalation:

Mandatory checks:

1. Layer Stability Check
2. Mapping Integrity Check
3. Cross-Dimension Coupling Audit
4. Backward Compatibility Review
5. Governance Freeze Confirmation

No Phase transition without explicit approval.

---

# IV. Dual-Core Separation Rule (New in v1.1)

The system operates under Dual-Core Architecture:

### App Layer (Governance Core)

Allowed:

* Structural audits
* Policy decisions
* Freeze approvals
* Guardrail definitions
* High-level system evaluation

Prohibited:

* Algorithm derivation
* Runtime debugging
* Code-level analysis
* Numerical experimentation

---

### VS Code / CODEX Layer (Engineering Core)

Allowed:

* Implementation
* Runtime execution
* Numerical modeling
* Debug instrumentation
* Validation reporting

Prohibited:

* Reinterpretation of Mapping Canonical
* Modification of Archetype semantics
* Expansion of Phase scope without Gate
* Structural Canonical edits

All Canonical authority resides in App layer.

---

# V. Mirror Drift Guardrail v1.0 (New in v1.1)

Purpose:

Recognize that minor asymmetry may arise in deterministic multi-agent simulations due to micro-order effects and Lanchester amplification.

We do not require mirror error = 0.

We require:

### A. No Systemic Drift

Across fixed benchmark scenario set:

New version mirror metrics must not show systematic upward trend.

### B. Not Worsened Gate

For Gate-level approval:

Primary metric (multi-scenario matrix metric) must satisfy:

mirror_not_worsened = True

Small numeric variation within tolerance is acceptable.

### C. Dual Metric Strategy

Two symmetry metrics must be tracked:

1. Gate Metric (matrix / cross-combination metric)
2. Watch Metric (single-scenario time-series metric)

Only Gate Metric blocks Phase progression.

Watch Metric is observational.

### D. No Artificial Equalization

Engineering must not:

* Introduce artificial synchronous resolution
* Force mirrored batch damage
* Modify combat semantics solely to reduce mirror metric

Mirror preservation must not compromise model class.

---

# VI. Determinism Guardrail

All combat outcomes must be reproducible under identical seed and configuration.

Any introduction of:

* Non-deterministic iteration
* Randomized tie-break
* Hardware-dependent ordering

Requires explicit Governance authorization.

---

# VII. Backward Compatibility Guardrail

Where specified:

If parameter h=0 or feature disabled:

System must strictly reduce to previous canonical behavior.

Regression failure blocks Phase progression.

---

# VIII. Debug Lifecycle Rule (Formalized)

Debug instrumentation:

* Must be labeled `debug-only`
* Default OFF
* Listed in each Engineering report
* Removed or explicitly retained before Phase escalation

Debug code must not silently influence runtime behavior.

---

# IX. Parameter Governance Rule

Every new exposed parameter must include:

* Purpose (1 sentence)
* Observable effect
* Default value
* Safe range
* Classification (Personality / Structural / Internal constant)

No silent parameter introduction permitted.

---

# X. Validation Gate Submission Format (Mandatory)

Every Engineering submission must include:

1. Visualization status
2. Combat validation
3. Backward compatibility (`h=0` if applicable)
4. Determinism check
5. Mirror check
6. Jitter or boundary metrics

Omission invalidates submission.

---

# XI. Evolution Principle

Sandbox is exploratory.

Archetype dimensions are foundational but not absolute truth.

However:

Dimension addition/removal requires explicit structural justification and Gate approval.

Parameter implementation flexibility is allowed within guardrails.
