Status: Revised spec only. No runtime code changes authorized.

---

## 0) Scope and Non-Goals

This document defines the Phase V.1 FSR runtime specification boundary only.

FSR in this phase must not modify:

- targeting logic or tie-breakers;
- combat resolution and damage semantics;
- movement objective definition;
- personality parameter set and mapping files;
- frozen canonical and governance layers.

---

## 1) Core Variables and Semantics

Per fleet `f`, at tick `t`:

- `s_f(t)`: target formation scale (dimensionless, `1.0` = no scale change).  
- `kappa_f(t)`: normalized cohesion indicator in `[0, 1]`.  
- `fsr_strength`: FSR step-strength control parameter (dimensionless).

Required semantics:

- `fsr_strength` controls only per-tick relaxation step magnitude applied to `lambda_f`.
- `fsr_strength` is not a global velocity multiplier.
- `fsr_strength` is not a displacement override.
- `fsr_strength` is not a force magnitude.
- `fsr_strength` is not a projection modifier.

Raw relaxation coefficient:

`k_f_raw = fsr_strength * (0.5 + 0.5 * kappa_f)`

Raw scale update:

`lambda_f_raw = 1 + k_f_raw * (s_f - 1)`

Unambiguous execution interpretation:

- The update is a single-step linear relaxation toward `s_f`, then safety-clamped (Section 4).
- Therefore, runtime uses a bounded step each tick, not an unconstrained direct jump.

---

## 2) Definition of Current Formation Scale `R_cur,f`

FSR uses RMS radius about fleet centroid, computed from alive units in fleet `f`.

Let alive unit positions be `(x_i, y_i)`, `i = 1..n_f`, with `n_f >= 1`.

Centroid:

- `cx_f = (1 / n_f) * sum_i x_i`
- `cy_f = (1 / n_f) * sum_i y_i`

Current formation scale:

- `R_cur,f = sqrt( (1 / n_f) * sum_i [ (x_i - cx_f)^2 + (y_i - cy_f)^2 ] )`

Rationale:

- robust to local shell thinning;
- less outlier-sensitive than max radius;
- aligned with density-based scale logic.

---

## 3) Single-Pass Constraint (Mandatory)

For each fleet on each tick, FSR block must do exactly:

1. compute centroid once;
2. compute `lambda_f` once;
3. apply one scaling transform once;
4. hand control to existing projection stage.

Strictly prohibited:

- scaling again after projection;
- iterative multi-pass scaling loops;
- contraction-expansion cycling inside one tick.

FSR is first-order, single-step relaxation only.

---

## 4) Bounded Per-Tick Scaling (Safety Clamp)

Safety clamp is mandatory and applied to `lambda_f_raw`:

- `lambda_f = clamp(lambda_f_raw, 1 - delta_lambda, 1 + delta_lambda)`

Where:

- `delta_lambda > 0` is a bounded per-tick scale envelope parameter.

Purpose:

- prevent instantaneous collapse/expansion;
- keep numerical behavior stable under extreme inputs.

This clamp defines the allowed per-tick step magnitude regardless of `s_f` or `k_f_raw`.

---

## 5) Zero-Side-Effect Bypass Guarantee

When either condition is true:

- `FSR_ENABLED == false`, or
- `fsr_strength == 0`

Required behavior:

- skip FSR block completely;
- do not update any FSR state/cache variables;
- do not re-capture baseline references (including any `N0`-style reference) mid-run;
- do not introduce alternate centroid paths or math branches that affect baseline output.

Regression requirement:

- outputs must be bitwise identical to baseline with FSR absent.

---

## 6) Fleet Isolation Guarantee

FSR operates independently per fleet.

Required:

- centroid and scale computations use only units from the same fleet;
- no cross-fleet centroid mixing;
- no cross-fleet scale coupling or shared update terms.

---

## 7) Tick Insertion Point

FSR insertion point is within movement integration flow, before existing projection:

1. state snapshot (existing)
2. evaluate cohesion/target/utility (existing, unchanged)
3. movement integration pre-projection positions (existing)
4. **FSR single-pass scaling (this spec)**
5. existing spacing/projection stage (unchanged)
6. combat resolution and bookkeeping (unchanged)

No FSR logic is permitted in targeting or combat stages.

## 8) Whitelist and Denylist (Spec Phase)

Current phase is specification-only.

Allowed in this phase:

- FSR specification document updates only.

Denied in this phase:

- runtime code edits (`runtime/**`, `runtime_v0_1.py`);
- canonical/mapping/governance frozen layer edits;
- behavior-changing implementation commits.

---

## 9) Validation Gate for Future Code Phase

Before approving implementation output, minimum checks:

1. Determinism: identical inputs -> identical outputs per tick.
2. Bypass equivalence: `FSR_ENABLED=false` and `fsr_strength=0` are bitwise-equal to baseline.
3. Single-pass conformance: one centroid, one lambda, one scaling per fleet per tick.
4. Safety clamp conformance: `lambda_f` never exceeds `[1-delta_lambda, 1+delta_lambda]`.
5. Fleet isolation: no cross-fleet data dependency in FSR calculations.
6. Regression stability: no new NaN/Inf or instability spikes under stress profiles.

---

## 10) Assumptions (Explicit)

1. `s_f` producer remains an existing or separately-governed policy input and is not redefined here.
2. `kappa_f` remains normalized in `[0, 1]`.
3. `delta_lambda` final default/range may be finalized at implementation authorization, but clamp behavior itself is mandatory.
4. This revised spec must be approved before any runtime implementation work starts.
